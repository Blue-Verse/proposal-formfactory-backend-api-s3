핵심 요구사항 분석 (RFP 정밀 반영)

  • 6종 RDBMS 테이블 설계·영구 구축 — Orders(uuid PK·status 1~5/-1·phone IDX·attachments JSON), Pricing(Team/Center/Items 3단 트리·customGuideObj·isClosed), Reviews/Cases(PK=timestamp), Marquee, Config(ff_set_* Key-Value), Popup(신설). LocalStorage 의존도 0% 달성
  • localStorage 트랜잭션 100% 대체 REST API (25+ 엔드포인트) — fetch() 호출 지점에 1:1 치환. 기존 Base64(btoa) 암호화 인터셉터를 우회해 순수 JSON 평문으로 송수신(보안은 HTTPS 대체)
  • storage 이벤트 대체 라이브 동기화 — 브라우저 storage 이벤트 의존을 SSE 또는 GET /api/orders/stats 폴링으로 대체하여 대시보드 강제 새로고침(Live Sync) 구현
  • S3 Presigned URL 4단계 다이렉트 업로드 — 발급→Multipart 직접 전송(2GB)→Key 반환→POST /api/orders 시퀀스. 관리자 파일 sync/download-url/Hard Delete 연동
  • 폼 제출·현황 트래커 보존 바인딩 — POST /api/orders 200 OK 후 무통장 입금 kakao-modal 원형 호출 유지, shakeError 통제 순서 엄수. 트래커는 이름 가운데 글자 서버단 마스킹(홍*동) + 완료/취소 30일 경과 건 블라인드
  • 마스터 대시보드 통제 — 인증(login/password), GET /api/admin/orders(연-월-일 3단 트리·경량 메타데이터), 기간 필터(전체/일·주·월·연간), 상태 CRUD·복구·Hard Delete, 스마트 에디터 draft(get/put), 단가표 PUT /bulk(Replace)
  • 비즈뿌리오/알리고 알림톡·SMS — 템플릿 변수({이름}{금액}{일정}) 치환, 제출·상태변경·수임료 트리거 자동 발송, 알림톡 실패 시 LMS/SMS Fallback, SMS 장부 soft/restore/hard delete
  • 7대 방어 로직 — ①S3 Key Collision(UUID 강제주입) ②Rate Limiting/과금폭탄(IP·번호 일일 제한·reCaptcha) ③DB 동시성 Transaction Lock ④S3 고아파일 Cron GC 배치 ⑤UUID v4 브루트포스 타파 ⑥WebShell 차단(MIME 강제검증·Content-Disposition) ⑦50MB 페이로드 과부하 방지(Lazy Load) + HTTPS·SQL Injection·CORS 화이트리스트

제안 포인트

1) 20페이지 RFP를 완독하고 설계에 반영했습니다. AS-IS의 btoa/Base64 평문 암호화 인터셉터, 3단계 마법사·딥서치·노무계산기·10탭 마스터 대시보드 구조, Orders/Pricing 컬럼 매핑까지 파악하여 기존 화면 로직을 단 1줄도 깨지 않는 바인딩 설계를 준비했습니다. 본 과업의 최우선 평가지표인 "프론트엔드 파손 제로(콘솔 에러 0건)"를 검수 기준으로 삼습니다.

2) Presigned URL + UUID v4 난수화는 검증된 구현 패턴. 유사 프로젝트에서 S3 Presigned URL 다이렉트 업로드와 파일명 난수화(Key Collision 방어)를 운영 수준으로 구현했으며, 2GB 멀티파트 업로드와 MIME 강제검증·Content-Disposition을 통한 WebShell 차단까지 동일 방식으로 처리합니다.

3) 7대 방어 로직 전체가 기존 구축 경험 영역. 브루트포스 방어, Rate Limiting, 업로드 검증은 물론 DB 동시성 Transaction Lock(단가표 Bulk Update 중 폼 제출 Dirty Read 방어), Cron 배치(고아 파일 GC)까지 핀테크·전자결재 백엔드에서 다계층으로 구현·운영한 경험을 그대로 적용합니다.

4) 알림톡/SMS는 동일 제공사(Ppurio/비즈뿌리오) 연동 경험 보유. 템플릿 변수 치환, 이벤트 트리거 자동 발송, 그리고 알림톡 실패 시 LMS/SMS Fallback 우회 구조까지 운영해, 과금 폭탄 방어와 발송 안정성을 동시에 확보합니다.

5) 산출물 완성도(1순위)에 집중. 다수 인력으로 설계·개발을 병행하여 30일 일정을 유지하면서, GitHub 권한 이관·정밀 ERD·Swagger/Postman API 명세까지 잔금 지급 요건을 모두 충족하는 형태로 납품합니다.

PRD (Product Requirements Document)

과업지시서(RFP) 기반으로 백엔드 기능 범위를 정의하고 구현 기준을 명확히 합니다.

1. 프로젝트 배경 및 목적

행정사사무소용 스마트 서면 설계 플랫폼 FORMFACTORY(jyformfactory.com)는 프론트엔드(단일 HTML SPA · Vanilla JS · Tailwind)가 마스터본 수준으로 100% 완성되어 있으나, 모든 비즈니스 로직(위임 신청, 단가표, 관리자 10탭)이 브라우저 LocalStorage 기반으로 구동되어 영속성·다중 사용자·대용량 파일 처리에 한계가 있습니다. 또한 현재는 btoa(encodeURIComponent()) 기반 LocalStorage 평문 암호화 인터셉터가 적용되어 있습니다. 본 과업의 목적은 ① LocalStorage 의존도 0% 달성 및 클라우드 RDBMS 영구 구축, ② 기존 Base64 암호화 로직을 우회한 순수 JSON 평문 송수신(보안은 HTTPS로 대체), ③ S3 Presigned URL 다이렉트 업로드 아키텍처 도입으로, 기존 화면을 단 1줄도 변경하지 않고 정식 서비스로 전환하는 것입니다.

3. 기능 요구사항

3.1 데이터베이스 (6종 RDBMS 테이블)

  • [필수] Orders — uuid(PK, JR-YYMMDD-XXXX 프론트 생성값 보존), status TINYINT(1:입금대기~5:발송완료/-1:취소, IDX), phone(IDX), cash_receipt, fee/timerType, attachments(JSON, S3 Key 배열), company/bizNo/position, name/birth/gender, service/description(TEXT), deliveryMethod/Address
  • [필수] Pricing — Team/Center/Items 3단 트리(약 80개), customGuideObj(JSON, 다중 탭 가이드 통째 저장), isClosed(BOOLEAN, 금일 마감)
  • [필수] Reviews/Cases — PK=timestamp(BigInt, 프론트 삭제/수정 함수가 timestamp 키 사용), status(pending/approved)
  • [필수] Marquee(배지·텍스트), Config(ff_set_* Key-Value 전역 환경변수), Popup(신설, enable/title/mediaType/mediaUrl)

3.2 REST API 서버 (25+ 엔드포인트)

  • [필수] 초기화: GET /api/config(15종 설정)·/api/popup·/api/pricing(80 트리)
  • [필수] 조회: GET /api/marquee·/api/reviews?status=approved·/api/cases(36+)
  • [필수] 라이브 동기화: 브라우저 storage 이벤트 대체 — SSE 또는 GET /api/orders/stats 폴링으로 refreshUI() 트리거
  • [필수] 제출: POST /api/orders — DB INSERT 후 200 OK, 무통장 입금 kakao-modal 원형 호출 유지, shakeError 통제 순서 엄수
  • [필수] 트래커: GET /api/orders/track?phone= — 이름 가운데 글자 서버단 마스킹(홍*동), 완료/취소 후 30일 경과 건 응답 자동 제외

3.3 S3 대용량 파일 업로드 (4단계 시퀀스)

  • [필수] ①Presigned URL 발급 요청 → ②브라우저→S3 직접 Multipart 전송 → ③반환 Key를 attachments 배열 매핑 → ④POST /api/orders
  • [필수] 1회 최대 2GB(개별 1GB) 대용량 처리, WAS 메모리 과부하 차단
  • [필수] UUID 강제 주입 파일명(Key Collision 방어), 관리자 파일 sync/download-url/Hard Delete(S3 SDK 원본 파기)

3.4 마스터 대시보드 & 인증

  • [필수] 하드코딩 폐기 → POST /api/admin/login·PUT /api/admin/password 인증
  • [필수] GET /api/admin/orders — 연-월-일 3단 트리 렌더링용 목록 Array, 기간 필터(전체/일·주·월·연간)
  • [필수] 상태 CRUD(forceStatusChange/restoreOrder/hardDeleteOrder DELETE), 스마트 에디터 draft(GET·PUT /api/orders/{uuid}/draft), 단가표 PUT /api/pricing/bulk(전체 트리 Replace)
  • [필수] CMS(marquee/review/case, timestamp 키), Config/Popup 저장, SMS 장부 soft/restore/hardDelete

3.5 외부 연동 (알림톡/SMS)

  • [필수] 비즈뿌리오/알리고 알림톡 API 서버단 연동, 템플릿 변수({이름}{금액}{일정}) 치환
  • [필수] 트리거: 신규 제출·상태 변경(1~5단계)·수임료/결제 안내 시 자동 발송
  • [필수] 알림톡 실패(미가입·장애) 시 LMS/SMS 우회 발송(Fallback)

3.6 보안 (7대 방어 로직 — RFP §8 4대 + §9 3대)

  • [§8] ①S3 Key Collision 방어(UUID/timestamp 강제) ②알림톡/SMS 과금폭탄 방어(IP·번호 일일 제한·reCaptcha) ③DB 동시성 Transaction Lock(Bulk Update 중 Dirty Read 방어) ④S3 고아파일 Cron GC 배치
  • [§9] ⑤UUID v4 브루트포스 타파(JR-YYMMDD-XXXX 대입 공격 차단) ⑥WebShell 차단(MIME 강제 검증·Content-Disposition) ⑦50MB 페이로드 과부하 방지(경량 메타데이터 + Lazy Load)
  • [§10] HTTPS(SSL) 전구간, SQL Injection 방어, CORS 화이트리스트, Rate Limiting

4. 비기능 요구사항

항목요구사항
무결성(최우선)프론트엔드 파손 제로 — 3단계 마법사 트랜지션, 아코디언, 노무 계산기 등 기존 UI/UX에 깨짐·콘솔 에러 0건 (본 과업 최우선 평가지표)
호환성기존 Vanilla JS fetch() 통신 규약과 완전 호환, Base64 우회 순수 JSON 송수신 (프레임워크 전환 불가)
성능대용량 파일 서버 미경유(다이렉트 업로드), 대시보드 경량 메타데이터 + Lazy Load로 50MB 과부하 방지
동시성단가표 Bulk Update 중 폼 제출 Dirty Read 방어 — Transaction Lock/낙관적 동시성 제어
보안HTTPS, 인증/인가, MIME 검증, Rate Limiting(reCaptcha), UUID v4 난수화 키, SQL Injection·CORS 방어
이식성/문서화Docker 기반 AWS EC2/RDS/S3 배포, Swagger/Postman API 명세·정밀 ERD·GitHub 권한 이관

5. 기술 제약사항

  • 제공된 기존 소스코드(Vanilla JS)의 fetch() 통신과 1:1 바인딩 — React/Vue 등 프레임워크 전환 제안 불가
